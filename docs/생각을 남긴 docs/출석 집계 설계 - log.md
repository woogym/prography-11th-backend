## 출석 집계 설계 결정

### 선택: 매번 집계 쿼리 방식

출석 통계를 별도 테이블에 저장하지 않고, 조회 시점마다 집계 쿼리를 실행하는 방식을 선택했습니다.

### 판단 근거

**1) 낮은 변경 빈도**
- 출석 기록은 세션 당 1회만 생성되며, 이후 수정이 거의 발생하지 않을것으로 예상했습니다.
- 실시간 정합성이 중요하지만 캐싱이 필요할 만큼 빈번하게 변경되지 않을것으로 예상했습니다.

**2) 낮은 조회 빈도**
- 일정 목록 조회는 관리자 페이지에서만 발생
- 일반 회원은 출석 통계를 조회하지 않음
- 위와 같은 이유로 트래픽이 집중될 가능성 낮다고 판단했습니다.

**3) 인덱스 최적화 가능**
- `session_id` 컬럼에 인덱스 설정 시 빠른 집계 가능
- 한 세션당 출석 기록 수는 제한적 (기수 인원 수준)

**4) 확장 가능한 구조**
- 성능 이슈 발생 시 Redis 캐싱으로 대응 가능
- 통계 테이블 추가 없이도 충분한 성능 확보 가능
- 초기 복잡도를 낮추고, 필요 시점에 최적화 적용

### 결론
추후 성능 이슈시에 대응 할 수 있으며, 데이터 정합성과 구현 단순성을 확보하는 방법인 집계 쿼리 방식을 채택했습니다.
